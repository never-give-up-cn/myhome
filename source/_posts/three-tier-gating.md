你是否想过，你每天都在使用的 Claude Code，可能只是完整版本的冰山一角？

Anthropic 的 Claude Code CLI 工具中，逆向工程发现了一套精密的**三层门禁系统**，控制着数百个功能的可见性。源码中发现了 **88+ 个构建时 feature flags**、**500+ 个运行时 A/B 测试标记**，以及一套完整的身份门控机制。

这不是安全机制——这是产品发布策略。让我们一层层拆解。

<!-- more -->

## 为什么需要三层门禁？

想象一下：Anthropic 正在开发一个实验性的"AI 助手模式"（内部代号 KAIROS）。这个功能：

- 可能还不够稳定，不能面向所有用户
- 需要在不同用户群体中测试效果
- 可能只对内部员工开放

三层门禁就是为了解决这些问题而设计的。功能请求从用户到代码执行，要经过三道独立的关卡：

```
用户请求 → 第一层(构建时) → 第二层(运行时) → 第三层(身份) → 功能可用
```

每一层独立运作，一个功能可能受多层同时控制。

---

## 第一层：构建时 Feature Flags（编译时宏控制）

这是最外层的大门，在代码**打包阶段**就已经决定了功能的生死。

| 属性 | 说明 |
|------|------|
| **控制方式** | 编译时宏 (`bun:bundle`) |
| **决策时机** | 打包时，代码直接被 DCE（死代码消除）移除 |
| **粒度** | 全有或全无 |
| **标记数量** | 88+ |
| **逆向可见性** | 代码残留可见，但永远走 `false` 分支 |

如果某个功能在编译时被关闭，相关代码在打包阶段就被彻底移除了——用户根本不可能接触到。这意味着你下载的 Claude Code 版本中，大量代码根本就不存在。

**影响**：这是最彻底的控制方式。关闭的功能连代码都不会存在于你的二进制文件中。

---

## 第二层：运行时 GrowthBook A/B 测试（远程求值）

第二层在**运行时**发挥作用。GrowthBook 是一个功能开关平台，Claude Code 集成了它的 SDK（约 1156 行代码），允许远程控制功能开关。

| 属性 | 说明 |
|------|------|
| **控制方式** | GrowthBook SDK 远程求值 |
| **决策时机** | 启动时 + 定期刷新 |
| **粒度** | 按用户 / 设备 / 组织定向 |
| **标记数量** | 500+（以 `tengu_*` 为前缀） |

这意味着 Anthropic 可以：

- **灰度发布**：先给 5% 的用户开启某个功能，观察效果
- **定向测试**：按用户属性（地区、组织规模、使用时长）决定功能可见性
- **远程关闭**：如果发现 bug，不需要用户更新版本，远程关闭功能
- **A/B 实验**：同时测试多个版本，收集数据

这 500+ 个标记覆盖了大量未公开的实验性能力。你用的 Claude Code 和隔壁桌用的，可能压根就不是同一个产品。

---

## 第三层：身份门控（USER_TYPE）

这是最内层、也是最严格的关卡。它通过一个构建时常量 `USER_TYPE` 来控制，只有两个取值：

- **`ant`** — Anthropic 内部员工
- **`external`** — 外部用户

| 属性 | 说明 |
|------|------|
| **控制方式** | 构建时 `--define` 常量 |
| **决策时机** | 打包时，常量折叠 |
| **粒度** | 按构建版本 |
| **标记数量** | 1 个（两种取值） |

这个常量在源码中出现了 **60+ 次**，控制着大量仅限 Anthropic 内部员工使用的功能。这些功能包括：

- 内部调试工具
- 特权命令
- 未公开的 API 端点
- 开发中的实验性功能

如果你是 `external` 用户，这些代码即使编译进了你的版本，也永远不会执行。

---

## 三层串联：一个功能的完整旅程

文档中提到了一个具体例子——**KAIROS 助手模式**（一个尚未公开的 AI 对话模式）。

```
KAIROS 功能的启动条件：
  1. 构建时: feature('KAIROS') 必须为 true
  2. 运行时: tengu_kairos 实验标记必须命中
  3. 身份:   USER_TYPE 必须为 'ant'
```

只有当三层门禁全部放行，这个功能才能被用户使用。任何一个关卡不通过，功能就不可见。

---

## 这背后的产品哲学

三层门禁系统的设计体现了几个重要的产品理念：

**渐进式发布**：不搞"大爆炸式发布"。功能先内部用，再小范围测试，再逐步扩大到所有用户。每层门禁对应一个发布阶段。

**可观测性优先**：500+ 个 A/B 标记意味着每个功能的上线效果都是可量化、可对比的。不是凭感觉发布，而是用数据说话。

**安全退出的能力**：任何一层都可以独立关闭某个功能。如果 KAIROS 出了问题，不需要发版修复——远程关闭第二层的实验标记就够了。

**内部工具是最好的测试**：让员工最先使用新功能，既是一种激励机制，也是最好的真实场景测试。

---

## 对我们开发者的启发

虽然我们大多数人不做 AI 工具，但这种三层门禁的设计思路值得借鉴：

1. **Feature Flags 是基础设施**：不要把功能开关当作临时方案，它应该是一套长效的基础设施
2. **构建时 + 运行时结合**：构建时控制"能做什么"，运行时控制"谁能用"，两者互补
3. **版本差异化管理**：不同的构建版本（内部版 vs 公开版）可以有截然不同的功能集，而不需要维护两套代码

如文档所言，这套系统的目的"不是阻止逆向工程，而是让 Anthropic 在不同用户群体中渐进式测试和发布功能"。但在我们看来，它展现了一个成熟产品的精细化运营能力。
